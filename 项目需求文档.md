# ChatGPT 连接器项目需求文档

> 文档状态：初稿  
> 当前阶段：项目立项 / 需求定义  
> 适用对象：产品、设计、客户端研发、服务端研发、测试、运营  
> 更新日期：2026-07-11

## 1. 项目概述

### 1.1 项目名称

暂定名称：**ChatGPT 连接器**

正式产品名称、品牌名称、图标和域名尚未确定，可在发布前替换。本文统一使用“连接器”指代 Windows 客户端，使用“网关”指代服务端 AI API 网关。

### 1.2 项目背景

部分 Codex 用户在配置 AI 模型服务时，需要自行查找 API 地址、申请密钥、修改本地配置，并自行处理线路不稳定、上游限流或密钥失效等问题，使用门槛较高。

本项目提供一个 Windows 桌面程序。用户安装或解压后打开程序，通过一个主要开关即可完成 Codex 的网关接入。客户端负责安全地备份和修改用户本地 `.codex` 目录内的相关配置；服务端负责用户鉴权、模型请求转发、上游凭证调度、健康检测、故障切换、额度控制和用量记录。

用户在 Codex 中使用的是本平台签发的用户网关密钥，不会获得或接触服务端保存的上游密钥。

### 1.3 产品定位

本产品定位为：

> 面向 Windows Codex 用户的一键式 AI 模型连接工具，由服务端多凭证 AI API 网关提供统一接入、健康检测和自动容灾能力。

产品不是单纯的脚本启动器，也不是向用户分发上游密钥的工具。Windows 客户端是接入入口，服务端网关是核心服务。

### 1.4 项目目标

- 将 Codex 的 AI 服务配置过程简化为一次点击。
- 为每位用户生成独立、可撤销、可轮换的网关密钥。
- 对用户隐藏上游地址、账号和真实密钥。
- 在服务端自动选择健康的上游凭证。
- 上游凭证异常时自动降权、熔断和恢复。
- 在不破坏用户其他 Codex 设置的前提下修改配置。
- 提供一键关闭和恢复原配置的能力。
- 建立可审计的请求、用量、成本和路由记录。
- 为后续套餐、支付、更多模型和多上游扩展预留能力。

### 1.5 非目标

首个版本暂不以以下能力为目标：

- 不开发完整的 Codex 替代品。
- 不在客户端直接保存或调用上游真实密钥。
- 不允许客户端下载服务器返回的任意脚本并静默执行。
- 不实现复杂的 AI 自动选模或跨模型续写。
- 不同时支持 macOS、Linux 和移动端。
- 不在首版覆盖图像、视频、语音等全部模型类型。
- 不在首版建设复杂代理商或多级分销体系。

## 2. 名词定义

| 名词 | 定义 |
|---|---|
| 连接器 | 用户安装在 Windows 上的桌面客户端 |
| 网关 | 本项目部署在服务器上的统一 AI API 服务 |
| 用户网关密钥 | 平台为单个用户签发、供其访问本项目网关的凭证 |
| 上游 | 提供实际 AI 模型能力的合作服务商或中转服务 |
| 上游凭证 | 保存在服务器中、用于访问上游服务的 API 密钥或等效凭证 |
| 密钥中枢 | 管理上游、账号、凭证、模型能力、额度和健康状态的服务端模块 |
| 路由 | 为一次用户请求选择具体上游与上游凭证的过程 |
| 短期粘性 | 在短时间内优先让同一用户使用同一健康线路的策略 |
| 熔断 | 线路持续异常时暂时停止向其发送新请求的保护机制 |
| Codex 配置 | 用户 `.codex` 目录中影响模型提供方、API 地址和凭证引用的配置 |

## 3. 用户与使用场景

### 3.1 目标用户

- 想快速使用 Codex，但不熟悉 API 配置的普通用户。
- 已经使用 Codex，希望获得更稳定模型线路的开发者。
- 需要统一额度、统一密钥和简单状态展示的小团队。
- 后续可能接入的企业或批量授权用户。

### 3.2 核心用户故事

1. 作为新用户，我希望打开程序后只需一次点击，就能完成 Codex 接入。
2. 作为谨慎用户，我希望修改配置前能看到修改内容，并随时恢复原配置。
3. 作为使用者，我希望某条上游线路故障时尽可能不影响当前工作。
4. 作为使用者，我希望看到连接状态、网关密钥、剩余额度和基础用量。
5. 作为运营人员，我希望快速发现失效、限流或余额不足的上游凭证。
6. 作为运营人员，我希望能停用异常用户密钥，控制额度、并发和模型权限。
7. 作为财务或运营人员，我希望每次用户请求和上游尝试都有可核对的用量记录。

### 3.3 典型使用流程

```text
下载并启动客户端
        ↓
登录、输入激活码或自动创建账号
        ↓
客户端从服务器获取用户专属网关凭证
        ↓
用户点击“开启连接”
        ↓
检测 Codex 与配置文件
        ↓
展示并确认拟修改内容
        ↓
备份、修改、验证配置
        ↓
测试网关连接
        ↓
提示配置成功，用户开始使用 Codex
```

关闭流程：

```text
用户点击“关闭连接”
        ↓
客户端识别本程序管理的配置字段
        ↓
恢复开启前的字段值或移除新增字段
        ↓
验证配置并提示结果
```

## 5. Windows 客户端需求

### 5.1 支持范围

- 首版支持 Windows 10 和 Windows 11 的主流 64 位版本。
- 安装形态待确认：安装包或免安装单文件程序。
- 客户端应尽量使用普通用户权限运行。
- 仅在确有必要且明确说明原因时请求管理员权限。
- 应支持简体中文，后续预留多语言能力。

### 5.2 客户端主要状态

| 状态 | 含义 | 用户可执行操作 |
|---|---|---|
| 未登录 | 尚未建立用户身份 | 登录、注册或输入激活码 |
| 未配置 | 已登录，但未修改 Codex 配置 | 开启连接、复制配置说明 |
| 配置中 | 正在备份、写入和验证 | 等待，不允许重复操作 |
| 已连接 | 配置有效且网关可访问 | 关闭连接、复制密钥、查看用量 |
| 服务异常 | 本地配置存在，但网关检测失败 | 重试、诊断、恢复原配置 |
| 配置冲突 | 用户或其他程序修改了受管字段 | 查看差异、重新配置或保留现状 |
| 需要更新 | 当前客户端版本不再受支持 | 更新客户端或退出 |

### 5.3 一键开启连接

用户点击主开关后，客户端必须按以下顺序执行：

1. 检查用户是否具有有效登录状态或激活状态。
2. 获取或确认用户专属网关密钥。
3. 检查 Codex 是否安装，以及目标配置目录是否存在。
4. 读取目标配置文件，不读取或上传无关文件。
5. 解析现有配置，识别配置格式和版本。
6. 计算需要新增或修改的字段。
7. 首次修改前展示变更摘要。
8. 创建带时间、文件哈希和版本信息的备份。
9. 写入同目录临时文件。
10. 重新解析临时文件并验证必要字段。
11. 使用原子替换方式更新正式配置文件。
12. 调用网关的低成本测试接口验证连通性和凭证。
13. 成功后更新客户端状态；任一步失败则自动回滚。

禁止使用不解析配置结构的全文字符串替换方式。具体 Codex 配置字段须在技术验证阶段根据目标 Codex 版本确定，不在本需求文档中写死。

### 5.4 一键关闭与恢复

- 用户必须可以在客户端中关闭连接。
- 客户端只恢复自己管理过的字段，不应覆盖用户在开启后对其他字段所做的修改。
- 如果原文件已被外部修改，应执行三方比较：原始备份、客户端写入结果、当前文件。
- 能安全合并时自动恢复；存在冲突时向用户展示差异并提供选择。
- 应提供独立的“恢复原配置”入口，即使网关不可连接也可以使用。
- 恢复过程不依赖服务器在线。

### 5.5 配置备份管理

- 备份默认保存在目标配置目录下的本产品专用子目录，具体目录名待技术设计确定。
- 备份不得上传到服务器。
- 每条备份记录包含创建时间、原文件路径、原文件哈希、客户端版本和修改字段清单。
- 至少保留最近 5 次有效备份；清理策略可配置。
- 用户可以查看备份列表并恢复指定版本。
- 如果检测到目标文件中包含其他敏感信息，不在界面和普通日志中完整展示。

### 5.6 用户网关密钥

- 用户网关密钥由服务器生成，客户端不得自行伪造或生成可直接访问网关的密钥。
- 每位用户至少拥有一个独立密钥。
- 密钥应可复制、轮换、撤销和重新签发。
- 界面默认只展示部分字符，用户主动点击后才短暂完整显示。
- 密钥不得写入普通明文日志、崩溃报告或统计数据。
- 如 Codex 配置必须保存明文密钥，客户端应明确告知用户其本地保存位置和风险。
- 后续可支持使用系统凭据存储，具体取决于 Codex 的凭证读取方式。

### 5.7 连接诊断

客户端应能够区分并提示：

- Codex 未安装；
- 配置目录不存在；
- 配置文件格式不受支持；
- 配置文件不可写；
- 网关域名无法解析；
- TLS 连接失败；
- 用户密钥无效或已撤销；
- 账户余额不足；
- 模型暂时不可用；
- 客户端版本过旧；
- 上游整体服务异常。

错误信息应面向普通用户，同时提供可复制的诊断编号，不应直接展示上游密钥、内部地址或敏感堆栈。

### 5.8 自动更新

- 客户端应支持检查更新。
- 更新包必须具有数字签名并校验完整性。
- 正式版本不得从服务器下载任意脚本后静默执行。
- 强制更新仅用于严重安全问题或协议不兼容，应在界面说明原因。
- 更新失败不得破坏已存在的 Codex 配置和备份。

## 6. 客户端界面需求

### 6.1 设计原则

- 主界面只突出一个核心操作：开启或关闭连接。
- 默认界面信息少而明确，高级能力放入二级页面。
- 不使用“绝对稳定”“永不掉线”等无法兑现的描述。
- 状态、错误和费用信息必须透明。
- 对配置文件的修改必须可解释、可恢复。

### 6.4 首次修改确认弹窗

弹窗应显示：

- 将修改哪个文件；
- 将修改哪些配置类别，例如 API 地址、提供方和凭证引用；
- 原配置将保存到哪里；
- 关闭连接后可以恢复；
- 不会上传用户原有配置内容；
- “确认修改”和“取消”按钮。

高级用户可以展开查看脱敏后的字段级变更，但不得显示现有完整密钥。

### 6.5 设置页面

首版建议包含：

- 开机启动开关；
- 自动检查更新；
- 客户端日志级别；
- 备份管理；
- 自动测试连接；
- 账户与退出登录；
- 隐私政策、服务条款和版本信息；
- 导出脱敏诊断报告。

暂不在首版向普通用户暴露具体上游密钥或复杂路由参数。

## 7. 服务端网关需求

### 7.1 API 兼容性

- 首版优先提供 Codex 当前目标版本可使用的 OpenAI 兼容接口。
- 具体需要兼容的路径、请求字段、流式事件和错误格式，应通过真实 Codex 客户端验证后冻结。
- 网关内部使用统一请求对象，再由上游适配器转换为上游协议。
- 对无法无损支持的参数应明确返回“不支持”，不得静默忽略关键字段。
- 网关应支持流式输出，并正确处理客户端主动断开。

### 7.2 用户鉴权

- 用户网关密钥具有唯一 ID、状态、创建时间和所属用户。
- 服务端不以明文形式长期保存完整用户密钥；应保存安全哈希或使用可验证令牌方案。
- 支持启用、停用、轮换和过期时间。
- 支持为密钥配置可用模型、额度、并发和速率限制。
- 所有请求必须关联明确的用户和密钥 ID。

### 7.3 限流与预算

至少支持以下限制：

- 每用户每分钟请求数；
- 每用户并发数；
- 每分钟输入与输出 Token 数；
- 单次请求最大上下文和最大输出；
- 每日或每月金额预算；
- 单次请求最大预估成本；
- 平台全局成本和并发保护阈值。

达到限制时返回统一、可识别的错误，不得继续消耗上游额度。

### 7.4 模型目录与映射

系统应维护：

- 对外模型名称；
- 上游实际模型名称；
- 所属上游；
- 上下文长度；
- 是否支持流式、工具调用、图片和结构化输出；
- 输入、输出及其他计费单价；
- 是否开放给用户；
- 是否允许跨线路切换；
- 可用地区和数据处理说明。

对外模型名称与实际模型之间的映射必须可审计。未经用户策略允许，不得将请求静默降级到不同能力模型。

### 7.5 路由策略

首版采用规则式动态路由，不使用机器学习路由。

候选筛选顺序：

1. 支持目标模型和请求能力；
2. 上游及凭证处于可用状态；
3. 未达到额度、限流或并发上限；
4. 满足用户套餐和地区要求；
5. 满足是否允许跨上游或跨模型切换的策略；
6. 按健康度、当前负载、延迟和成本计算权重；
7. 选择得分较高的候选线路。

首版建议支持短期粘性，但不建立用户密钥与上游凭证的永久绑定。粘性时间建议可配置，初始值可在 5 至 30 分钟内通过内测确定。

### 7.6 重试和故障转移

- 请求尚未向用户输出内容时，可在可恢复错误下切换另一健康线路。
- 单个用户请求首版最多进行 2 次上游尝试。
- 已经开始流式输出后，默认不自动换线路续写。
- 用户参数错误、内容策略拒绝和不支持的能力不得通过换线路反复尝试。
- 工具调用或其他可能产生副作用的阶段不得盲目重试。
- 每次尝试都必须生成独立记录，并归属于同一用户请求 ID。
- 请求必须具有连接超时、首字节超时、流式空闲超时和总截止时间。

具体超时值按模型类型配置，不在本阶段统一写死。

### 7.7 流式请求

- 支持标准流式事件透传或转换。
- 首个输出事件前失败可执行有限故障转移。
- 输出开始后发生异常，应返回网关定义的结束或错误事件，并记录流式中断。
- 用户断开后应尽快取消上游请求，避免继续计费；无法取消时记录额外成本。
- 统计首字节时间、完整响应时间和流式中断率。

## 8. 密钥中枢需求

### 8.1 上游管理

每个上游应保存：

- 上游名称和内部编号；
- API 基础地址；
- 协议适配器类型；
- 启用状态；
- 支持模型；
- 价格及价格生效版本；
- 总体并发和速率限制；
- 联系人与故障沟通方式；
- 数据处理及商业授权备注；
- 最近健康状态和维护信息。

### 8.2 上游凭证管理

每个凭证应保存：

- 凭证 ID，不在管理列表直接显示完整值；
- 所属上游和账号；
- 加密后的凭证材料；
- 可用模型和权限；
- 启用、停用、熔断、余额不足等状态；
- 限流和并发参数；
- 当前负载；
- 最近成功时间；
- 最近错误及分类；
- 最近窗口的成功率和延迟；
- 余额或预算信息；
- 创建、轮换和过期时间。

上游凭证不得下发到客户端，也不得出现在普通日志、错误消息和前端接口中。

### 8.3 健康状态机

建议状态：

```text
正常 Healthy
   ↓ 指标轻微异常
降权 Degraded
   ↓ 连续异常或超过阈值
熔断 Open
   ↓ 冷却期结束
试探 Half-open
   ↓ 探测成功
正常 Healthy
```

另设需要人工处理的状态：禁用、认证失败、余额不足、已过期。

一次偶发超时不得永久移除凭证。熔断时间和恢复条件应可配置，并可采用递增冷却时间。

### 8.4 健康检测方式

主动检测：

- 定期执行低成本模型探测；
- 验证连接、认证、目标模型和响应格式；
- 控制频率，防止探测本身造成显著成本。

被动检测：

- 使用真实请求统计成功率、首字节延迟和中断率；
- 按错误类型更新凭证、模型或上游层级的健康指标；
- 用户参数错误和内容拒绝不计入线路故障率。

需要分别维护：上游整体健康、具体模型健康、具体凭证健康。

### 8.5 错误分类与动作

| 错误类型 | 默认动作 |
|---|---|
| 网络连接失败或超时 | 降权，达到阈值后熔断 |
| 上游 429 | 临时冷却并降低权重 |
| 上游 5xx | 同时更新上游与凭证健康指标 |
| 401/403 | 立即隔离凭证并告警 |
| 余额不足 | 停止新请求并通知运营 |
| 指定模型不存在 | 移除该凭证对应模型能力并告警 |
| 响应格式异常 | 降权，连续发生后熔断 |
| 用户参数错误 | 返回用户，不改变凭证健康度 |
| 内容安全拒绝 | 返回用户，通常不改变凭证健康度 |

## 9. 管理后台需求

### 9.1 仪表盘

- 当前请求量、并发量和成功率；
- P50、P95、P99 延迟；
- 流式中断率；
- 各上游健康状态；
- 可用、降权、熔断和停用凭证数量；
- 今日用户计费、上游成本和估算毛利；
- 429、5xx、认证失败和余额不足告警；
- 当前影响用户的服务事件。

### 9.2 用户管理

- 搜索和查看用户；
- 查看用户状态、套餐、额度和基础用量；
- 签发、撤销和轮换用户网关密钥；
- 配置模型权限、限流和预算；
- 冻结异常用户；
- 查看脱敏后的请求元数据和操作审计；
- 执行额度补偿并记录原因。

### 9.3 上游与凭证管理

- 新增、编辑、启用和停用上游；
- 添加、轮换和停用凭证；
- 查看凭证健康、负载、余额和错误；
- 人工触发健康测试；
- 调整权重、并发、限流和熔断参数；
- 批量停用某个上游或模型；
- 管理模型映射和价格版本；
- 所有敏感操作必须写入审计日志。

### 9.4 请求与账单查询

一次用户请求与多次上游尝试应分开记录：

```text
用户请求
├── 尝试 1：上游 A / 凭证 12 / 超时 / 可能产生成本
└── 尝试 2：上游 A / 凭证 27 / 成功 / 产生成本
```

后台应支持按请求 ID、用户、模型、上游、凭证、状态和时间查询。默认不展示用户完整提示词和模型输出。

### 9.5 权限控制

至少区分：

- 超级管理员；
- 运维人员；
- 客服人员；
- 财务或运营人员；
- 只读审计人员。

不同角色只能访问完成工作所需的最少信息。查看、复制或更新上游凭证属于高风险操作，应进行二次验证或审批。

## 10. 数据与计费需求

### 10.1 核心数据实体

- 用户；
- 用户会话或设备；
- 用户网关密钥；
- 套餐与用户权益；
- 模型目录；
- 上游；
- 上游账号；
- 上游凭证；
- 路由策略；
- 用户请求；
- 上游请求尝试；
- 用量记录；
- 账本流水；
- 配置版本；
- 健康检测记录；
- 告警与服务事件；
- 管理操作审计。

### 10.2 用量记录

每次用户请求至少记录：

- 用户请求 ID；
- 用户和用户密钥 ID；
- 对外模型；
- 请求开始、首字节和结束时间；
- 最终状态和错误分类；
- 输入、输出、缓存及其他可用 Token 数据；
- 用户计费金额；
- 是否发生重试；
- 客户端版本和脱敏来源信息。

每次上游尝试至少记录：

- 上游尝试 ID；
- 所属用户请求 ID；
- 上游、模型和凭证 ID；
- 开始、首字节和结束时间；
- 状态、HTTP 状态和内部错误分类；
- 上游报告用量；
- 估算成本和最终对账成本；
- 是否向用户输出过内容。

### 10.3 账本

如果产品采用余额计费，金额变化应通过不可变流水记录：

- 充值；
- 请求扣费；
- 失败退款；
- 活动赠送；
- 人工补偿；
- 退款；
- 管理调整。

余额由流水汇总或可靠的派生余额得到。任何人工调整必须记录操作人和原因。

### 10.4 对账

- 支持网关用量与上游账单的周期性核对。
- 记录重试造成的额外上游成本。
- 上游数据延迟时先记录估算值，后续更新对账状态，不覆盖原始历史。
- 当成本异常、价格未配置或毛利低于阈值时告警。

## 11. 安全与隐私要求

### 11.1 基本原则

- 上游凭证仅存在于服务器端受控环境。
- 用户原 Codex 配置不上传服务器。
- 默认不保存完整提示词和模型输出。
- 日志、指标和错误报告只保留必要的脱敏元数据。
- 所有网络请求使用 TLS。
- 管理后台必须使用强认证，并建议支持多因素认证。

### 11.2 凭证安全

- 上游凭证使用密钥管理服务或等效机制加密。
- 数据库和应用日志不得出现明文上游凭证。
- 凭证解密权限遵循最小权限原则。
- 支持凭证轮换，轮换过程中不中断已有请求。
- 泄露时能够快速定位影响范围并停用。

### 11.3 客户端安全

- 可执行程序和更新包应数字签名。
- 配置写入前备份，写入后验证，失败自动回滚。
- 不从服务器接收任意命令并在本地执行。
- 不收集用户项目文件、命令历史或现有第三方密钥。
- 崩溃报告发送前应脱敏并征得用户同意。
- 防止多个客户端进程同时修改同一配置文件。

### 11.4 服务端风控

- 用户级限流、并发和预算控制；
- 检测密钥从大量设备或异常地区同时使用；
- 防止用户密钥被公开或转售后无限消耗；
- 限制单次请求体、上下文和输出大小；
- 为上游传递不可逆的匿名下游用户标识，前提是上游支持；
- 具有全局紧急停服、上游停用和模型停用开关；
- 对异常高成本行为实时告警。

### 11.5 用户透明度

正式运营前应准备：

- 服务条款；
- 隐私政策；
- 数据保留政策；
- 上游或分处理方说明；
- 退款和服务中断规则；
- 可接受使用政策；
- 用户数据删除和账号注销方式。

## 12. 非功能性需求

### 12.1 可用性

- 网关目标可用性 SLA 待内测数据和上游承诺确定。
- 首版内部目标建议以成功率、首字节延迟和流式中断率分别衡量，而不是只使用单一可用率。
- 核心服务应具备健康检查、自动重启、熔断和告警。
- 单个凭证失效不得导致整个服务停止。

### 12.2 性能

- 网关自身增加的非模型处理延迟应尽量低，并持续监控。
- 流式数据不得在网关中无必要地完整缓冲。
- 客户端主界面应快速启动，配置操作过程应提供进度反馈。
- 具体并发和延迟指标在压测阶段确定。

### 12.3 可扩展性

- 新增上游应通过适配器完成，不应大量修改核心鉴权与计费逻辑。
- 模型、价格、限流和路由参数应配置化。
- 首版可以采用模块化单体，避免过早拆分大量微服务。
- 用户鉴权、请求记录和计费数据结构需为后续扩容预留空间。

### 12.4 可观测性

需要监控：

- 请求量、成功率、错误率；
- 各阶段延迟；
- 流式连接数和中断率；
- 上游、模型及凭证健康；
- 用户与平台并发；
- Token 与金额消耗；
- 重试率和重试损耗；
- 数据库、缓存和队列状态；
- 客户端版本分布和配置失败率。

所有请求应具有贯穿网关和上游尝试的追踪 ID。

### 12.5 可维护性

- 开发、测试和生产环境严格隔离。
- 上游凭证不得复用于非生产测试。
- 配置变更应有版本和回滚能力。
- 数据库结构变更应可迁移和回滚。
- 发布前完成自动化测试、灰度发布和回滚预案。

## 13. 异常与边界场景

系统设计和测试必须覆盖：

- 用户未安装 Codex；
- `.codex` 目录不存在；
- 配置文件为空、损坏或权限不足；
- 配置结构因 Codex 升级发生变化；
- 用户手动修改了客户端管理的配置字段；
- 客户端在写配置时断电或崩溃；
- 多个客户端实例同时修改配置；
- Windows Defender 或其他安全软件拦截；
- 网关密钥过期、撤销或余额不足；
- 上游凭证认证失败；
- 某个模型下线或更名；
- 上游整体不可用；
- 上游返回成功但网关等待超时；
- 流式输出中途断开；
- 用户主动取消请求；
- 同一用户密钥出现异常高并发；
- 上游价格改变但本地价格尚未更新；
- 客户端版本与网关协议不兼容；
- 网关不可用时用户需要离线恢复配置。

### 14.1 MVP 必须完成

客户端：

- Windows 10/11 支持；
- 用户登录、激活或其他最小身份机制；
- 获取用户专属网关密钥；
- 检测 Codex 配置；
- 首次修改确认；
- 配置备份、结构化修改、验证和原子替换；
- 开启、关闭和恢复原配置；
- 网关连通性测试；
- 基础状态和错误提示；
- 脱敏诊断报告。

服务端：

- 用户鉴权；
- 至少一个 OpenAI 兼容入口；
- 文本与流式文本；
- 固定模型映射；
- 至少一家合作上游的多个凭证，架构预留第二上游；
- 动态加权路由和短期粘性；
- 主动与被动健康检测；
- 降权、熔断、试探恢复；
- 首个输出前最多一次故障转移；
- 用户级额度、速率和并发限制；
- 用户请求及上游尝试记录；

- 自动跨模型降级；
- 图像、视频和语音模型；
- 多级代理商体系；

- macOS 和 Linux 客户端；

- 自动工具执行；
- 对用户开放精细的上游选择。

## 15. 验收标准

### 15.1 客户端核心验收

1. 在受支持的干净 Windows 环境中，用户可完成安装或启动。
2. 点击开启后可以正确检测目标 Codex 配置。
3. 修改前生成有效备份，且原文件内容可恢复。
4. 修改失败或进程异常后，不留下无法解析的正式配置文件。
5. 不覆盖与本产品无关的用户配置。
6. 用户可以在没有网络的情况下恢复原配置。
7. 客户端日志不包含完整用户密钥、原有第三方密钥或上游密钥。
8. 网关不可访问时给出准确提示和诊断编号。
9. 重复点击或多进程运行不会同时破坏配置。

### 15.2 网关核心验收

1. 无效、撤销和超额用户密钥不能调用上游。
2. 用户永远无法通过响应、日志或错误信息获得上游凭证。
3. 健康凭证能够正常完成普通及流式请求。
4. 凭证达到故障阈值后自动熔断，不再接收正常流量。
5. 冷却后只接收试探流量，成功后再恢复。
6. 首次尝试在输出前发生可恢复故障时，可切换另一健康凭证。
7. 已开始流式输出后不进行不可控的静默续写。
8. 每个用户请求和每次上游尝试均可通过追踪 ID 查询。
9. 用户限流、并发、额度及平台总预算保护有效。
10. 管理员可快速停用单个凭证、上游、模型或用户。

### 15.3 安全验收

1. 上游凭证在数据库和备份中均为加密状态。
2. 管理后台未经授权不能查看或修改敏感数据。
3. 更新包被篡改时客户端拒绝安装。
4. 客户端不会执行服务器下发的任意脚本或命令。
5. 日志、监控和错误报告通过敏感信息扫描。
6. 针对用户密钥滥用、重放、高并发和超大请求进行测试。

## 16. 建议实施阶段

### 阶段 0：技术验证

- 确定目标 Codex 版本和准确配置结构；
- 验证自定义网关地址、用户凭证和模型配置方式；
- 验证普通请求、流式请求及错误格式；
- 与上游确定接口、模型、价格、限流和合作方式；
- 建立最小网关转发原型。

### 阶段 1：内部 MVP

- 完成单用户端到端闭环；
- 完成配置备份、修改、恢复；
- 完成网关鉴权、上游调用和基础记录；
- 完成基础健康检测和人工停用。

### 阶段 2：小规模内测

- 邀请约 10 至 30 名用户；
- 验证配置兼容性和恢复可靠性；
- 收集真实成功率、延迟、流式中断和成本数据；
- 加入动态路由、熔断、限流和告警；
- 完成隐私政策和服务条款初版。

### 阶段 3：公开测试

- 建立支付或套餐体系；
- 完善管理后台、对账和客服工具；
- 客户端签名与自动更新；
- 状态页面和故障公告；
- 灰度扩大用户规模。

### 阶段 4：正式运营

- 明确 SLA 与退款规则；
- 扩展第二独立上游；
- 完成容量规划、灾备和安全评估；
- 根据实际需求扩展模型、平台和企业能力。

- 是否自动写入密钥，还是默认仅复制给用户手动粘贴；
- 自动更新服务与代码签名证书方案。

### 服务端

- 首版部署云厂商和地区；
- 首批上游数量和是否具备独立故障域；
- 上游能否提供余额、用量和故障通知接口；
- 具体模型目录、价格与映射；
- 具体超时、熔断和短期粘性参数；
- 是否向用户开放线路策略；
- 是否保存脱敏提示词摘要用于争议处理。

### 合作与合规

- 与上游的书面合作与转售授权形式；
- 上游数据保留和训练政策；
- 上游能否接收匿名下游用户标识；
- 用户内容和业务数据的保存周期；
- 服务地区、年龄要求和内容政策；
- 财务结算、发票与税务方案。

## 18. 关键项目原则

1. **用户密钥与上游凭证严格分离。** 用户只能持有平台签发的网关凭证。
2. **客户端修改必须可解释、可验证、可恢复。** 网关离线时仍能恢复原配置。
3. **路由不采用纯随机和永久绑定。** 使用能力过滤、健康评分、负载权重和短期粘性。
4. **健康异常不等于永久失效。** 使用降权、熔断、试探恢复的状态机。
5. **流式输出后不进行不可控的静默切换。** 防止输出矛盾或重复执行。
6. **默认不收集用户业务内容。** 仅保留运行、计量和审计所需的脱敏元数据。
7. **任何请求都能追踪到用户及具体上游尝试。** 同时避免向上游暴露用户真实身份。
8. **上游合作关系需产品化。** 明确授权、价格、限流、故障沟通、数据和结算规则。
9. **先完成小规模可靠闭环，再扩展功能。** 稳定性、成本控制和恢复能力优先于模型数量。
